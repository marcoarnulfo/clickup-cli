# v1.7 — Billing depth (moat) · Design

> Spec di progetto (in `docs/superpowers/specs/`). Scritto in italiano; codice e
> identificatori in inglese. Deriva dalla milestone GitHub **#5 "v1.7 — Billing
> depth (moat)"** ed è stato rivisto da un agente fable (i fix Critical/Important
> della review sono incorporati qui).

## 1. Contesto & obiettivo

La v1.6 ha posato le fondamenta (rebrand `clup`, `internal/service`, rate limiter,
report headless). La v1.7 costruisce **il moat**: profondità di fatturazione che
nessun competitor ClickUp CLI offre in una TUI. Portiamo il dominio da
`amount = ore × rate_lista` a un motore di pricing con **split billable/non-billable,
tariffe per-membro e per-(lista,membro), arrotondamento a incremento di
fatturazione, subtotali multi-valuta (senza conversione FX), budget & burn-down**,
più **correttezza timezone/DST**, **report settimanale**, **tag come categorie**,
e **export "fattura non fiscale" HTML + CSV per tool di fatturazione**.

**Fatturazione fiscale ESCLUSA** (IVA/numerazione/PDF legale): teniamo solo il
valore neutro e paese-agnostico.

## 3. Principio di purezza (invariante)

`internal/report` e `internal/duration` restano **PURI**: nessun I/O, nessun
`time.Now`, nessun import di `config`/`clickup`. `report` **può** importare
`duration` (entrambi puri). Tutta la matematica di pricing/rounding/budget/valuta
è funzione pura con input espliciti (incluso `*time.Location` e le date; **mai**
un orologio implicito). La review ha confermato: nessuna violazione strutturale di
purezza nel design; i punti di collocazione (rounding, budget) sono risolti sotto.

## 4. Modello di dominio (pure) — `internal/report`

### 4.1 `TimeEntry`
Aggiunta:
```go
Billable bool // #51 — da `billable` nella GET time_entries
```
Decodifica lato client come `*bool` per distinguere "campo assente" da `false`
(vedi §12). Nel modello puro il campo è già risolto a `bool`.

### 4.2 Pricing (sostituisce l'uso diretto di `Rates` in `Build`) — fix review C4
`Rates` **non** diventa una God-struct. Separiamo:
```go
type ListMember struct {
    ListID string
    UserID int
}

type Rates struct {
    Default      float64
    ByList       map[string]float64      // listID -> rate
    ByMember     map[int]float64         // userID -> rate
    ByListMember map[ListMember]float64  // (listID,userID) -> rate
}

// For applica la precedenza: (lista,membro) > membro > lista > default.
func (r Rates) For(listID string, userID int) float64

type Pricing struct {
    Rates           Rates
    Currencies      map[string]string // listID -> currency ISO (es. "EUR")
    DefaultCurrency string            // fallback quando la lista non è in Currencies
    Rounding        RoundRule         // §5
}
```
`Currencies`/`Rounding` vivono in `Pricing` (ciò che trasforma ore in denaro), **non**
in `Rates`. **`Budgets` sta FUORI** da `Pricing` (non è pricing): è input separato di
`BudgetLines` (§8).

### 4.3 Bucket & Report — fix review C1 (correttezza valuta)
Un bucket per `GroupByTask`/`Day`/`Member`/`Tag`/`Total` può aggregare liste con
**valute diverse**: un `Amount` scalare singolo è quindi errato. Modello:
```go
type CurrencyAmount struct {
    Currency string  `json:"currency"`
    Amount   float64 `json:"amount"`
}

type Bucket struct {
    Label         string           `json:"label"`
    Key           string           `json:"key"`            // ID stabile (listID/taskID/userID/day/tag)
    Hours         float64          `json:"hours"`          // tutte le ore (billable+non), raw
    BillableHours float64          `json:"billable_hours"` // raw, solo entry billable
    BilledHours   float64          `json:"billed_hours"`   // billable dopo rounding (§5) — base dell'importo
    Amounts       []CurrencyAmount `json:"amounts"`        // subtotali per valuta dentro il bucket
}

type CurrencySubtotal struct {
    Currency      string  `json:"currency"`
    Hours         float64 `json:"hours"`
    BillableHours float64 `json:"billable_hours"`
    BilledHours   float64 `json:"billed_hours"`
    Amount        float64 `json:"amount"`
}

type Report struct {
    Start, End       time.Time
    Timezone         string   // nome IANA della location usata (§6) — nel JSON
    Scope, GroupBy   string
    DefaultCurrency  string
    Buckets          []Bucket
    TotalHours       float64
    BillableHours    float64
    NonBillableHours float64
    BilledHours      float64
    CurrencySubtotals []CurrencySubtotal // verità per gli importi (no totale cross-valuta)
    TotalAmount      float64  // valorizzato SOLO se il report è mono-valuta; altrimenti 0
    // Rate/Currency singoli RIMOSSI dal Report (erano assunzioni mono-tariffa/valuta).
}
```
- **Bucketing per ID, non per nome** (fix C1): `keyFor` usa l'ID (`ListID`, `TaskID`,
  `UserID`, giorno-in-tz, tag) come `Key`, e un nome leggibile come `Label`. Elimina
  la fusione di liste omonime con rate/valute diverse. Il `Filter` per lista (oggi su
  `ListName`) va allineato a matchare per ID dove tocca la correttezza valuta.
- `TotalAmount` è definito nel JSON come: valorizzato **solo se** `len(CurrencySubtotals) == 1`,
  altrimenti `0` e la verità è in `CurrencySubtotals`. Documentato nello schema.

### 4.4 Firma di `Build` — fix review C4
```go
func Build(entries []TimeEntry, groupBy string, p Pricing,
    start, end time.Time, loc *time.Location) Report
```
Niente `estimates` (#68 fuori), niente `budgets`, niente `now` nel dominio.
Budget è funzione separata (§8). `loc` è esplicito (§6).

## 5. Rounding — fix review C3, I1, I2

### 5.1 Meccanica pura in `internal/duration`
```go
type RoundMode int
const ( RoundNearest RoundMode = iota; RoundUp )

// Round arrotonda d al multiplo di increment secondo mode.
// increment <= 0 => ritorna d invariato.
func Round(d, increment time.Duration, mode RoundMode) time.Duration
```
`duration` espone **solo** la meccanica (nessun concetto di "giorno"/"entry").

### 5.2 Regola completa in `internal/report`
```go
type RoundScope int
const ( PerEntry RoundScope = iota; PerDay )

type RoundRule struct {
    Increment time.Duration       // 0 => off (default)
    Mode      duration.RoundMode
    Scope     RoundScope
}
```
Grana formale (fix C3), per eliminare ambiguità tra i subagent:
- Si arrotondano **solo le ore billable**. Le non-billable **non** si arrotondano mai
  (non producono importo) e restano in `Hours`/`NonBillableHours` grezze.
- L'arrotondamento opera sulla **chiave più fine che condivide una rate**:
  - `PerEntry`: ogni entry billable arrotondata singolarmente.
  - `PerDay`: somma delle durate billable per `(giorno-in-tz, listID, userID)`,
    poi `Round(...)`. Il "giorno" è calcolato in `loc` (§6); nei giorni DST (23/25h)
    resta un normale confine di calendario in `loc`, non 24h fisse.
- Campi risultanti espliciti: `Hours` (raw, tutte), `BillableHours` (raw),
  `BilledHours` (arrotondate, base dell'`Amount`).

### 5.3 Ledger monetario unico (fix I1)
Una sola regola in tutta la pipeline, per garantire **totali identici tra JSON, CSV
invoice e HTML**:
> L'importo si arrotonda a 2 decimali all'unità rate-omogenea più piccola
> (`PerEntry` → per entry billable; `PerDay` → per gruppo `(giorno,lista,membro)`),
> e ogni aggregato è la **somma di importi già arrotondati** a quel livello.

Le righe del CSV invoicing (§10) sono **esattamente** queste unità. Test cross-export
obbligatorio: `sum(righe CSV) == CurrencySubtotals JSON == totali HTML`.

## 6. Timezone / periodi — fix review C2, issue #72

### 6.1 Range parametrici in `internal/report/period.go`
```go
func MonthRange(year int, month time.Month, loc *time.Location) (start, end time.Time)
func WeekRange(isoYear, isoWeek int, loc *time.Location) (start, end time.Time) // §7
func CustomRange(from, to time.Time, loc *time.Location) (start, end time.Time)
```
`loc == nil` ⇒ **UTC** (retro-compat). Le entry si assegnano al giorno/mese in base
a `Start` **convertito in `loc`**. Half-open `[start, end)` invariato.

### 6.2 Regola a due binari (fix C2, la parte cruciale)
- **TUI**: usa la location risolta da config (`timezone` IANA) o, se assente,
  `time.Local`. Coerente con l'ambiente dell'utente.
- **Headless (`clup report`)**: **UTC di default** — il comportamento attuale non
  cambia mai di nascosto. Si passa a un'altra location **solo** se esplicitata via
  `timezone` in config **o** flag `--tz <IANA>`. Questo preserva la riproducibilità
  dello "schema JSON stabile" tra macchine/CI.
- In **ogni** caso l'output JSON dichiara la location usata (`Report.Timezone`) e
  aggiunge un `schema_version` all'oggetto (lo schema evolve: additivo, i campi
  nuovi non rompono i parser esistenti; **`rate`/`currency` restano deprecati**, non
  rimossi). Nota: `Report.Timezone = loc.String()` vale `"UTC"` (headless) o il nome
  IANA se configurato; una TUI **senza** `timezone` in config emette `"Local"`
  (`time.Local.String()`), non un nome IANA — limite accettato, si consiglia di
  configurare un `timezone` esplicito.

### 6.3 Risoluzione della location (impura, fuori dal dominio)
Helper in `internal/service` (o `internal/cli`): `LoadLocation(name string) (*time.Location, error)`
via `time.LoadLocation`; nome vuoto ⇒ il default del binario (TUI: `time.Local`;
headless: `time.UTC`). Errore di IANA sconosciuto ⇒ messaggio chiaro, non panic.

## 7. Report settimanale — issue #4
- `WeekRange(isoYear, isoWeek, loc)` **ISO-8601** (settimana lun–dom; anno ISO ≠ anno
  civile ai bordi 29 dic–3 gen / settimana 53 → test espliciti).
- Il preset headless `this_week` **esiste già** in `period.go`: si aggancia a
  `WeekRange`, non si reinventa.
- TUI: toggle **month/week** nella home; il resto della pipeline è identico.
- Headless: flag `--week <ISOyear-Www>` (es. `2026-W30`) alternativo a `--month`.

## 8. Budget & burn-down — issue #64 (fix C4: funzione pura separata, no `now`)
```go
type BudgetLine struct {
    ListID      string  `json:"list_id"`
    ListName    string  `json:"list"`
    Currency    string  `json:"currency"`   // eredita la valuta della lista
    Budget      float64 `json:"budget"`
    Billed      float64 `json:"billed"`
    Remaining   float64 `json:"remaining"`
    PercentUsed float64 `json:"percent_used"`
}

// BudgetLines calcola le righe budget dal fatturato per-lista e dai budget
// configurati. Puro: nessun now (il "burn" usa le date delle entry già nel Report).
func BudgetLines(billedByList map[string]float64, budgets map[string]float64,
    currencies map[string]string, defaultCurrency string) []BudgetLine
```
`billedByList` deriva dai bucket per-lista del Report. Il budget di una lista **eredita
la valuta della lista** (dichiarato in config, §9). Vista TUI: barra di progresso
testuale semplice (i gauge veri arrivano in v1.9).

## 9. Config schema v2 — fix review I5
Bump `currentSchemaVersion` 1 → 2. Nuovi campi **opzionali**, retro-compatibili;
`migrate` porta v1→v2 senza perdite (campi nuovi vuoti). Blocco `billing:` per i
campi di fatturazione; `timezone` top-level (non è billing).
```yaml
schema_version: 2
timezone: Europe/Rome        # opzionale; TUI default time.Local, headless default UTC
billing:
  default_currency: EUR
  rates_by_member:           # userID -> rate
    123: 80
  rate_overrides:            # lista di oggetti, NO chiave composita "901,123"
    - { list: "901", member: 123, rate: 90 }
  currencies:                # listID -> valuta
    list_abc: EUR
    list_xyz: USD
  budgets:                   # listID -> importo (valuta = quella della lista)
    list_abc: 5000
  rounding:
    increment: 15m           # "0" o assente => off
    mode: nearest            # nearest | up
    scope: entry             # entry | day
```
- `rate` (default) e `rates` (per-lista) **esistenti restano** dove sono per
  compat; i nuovi per-membro/override/valute/budget/rounding stanno sotto `billing:`.
  (In alternativa, migrazione che sposta `rate`/`rates`/`currency` sotto `billing:`
  al primo save v2 — da decidere nel piano; il default è **non** spostare per minimo
  rischio.)
- Parsing `rate_overrides` come sequenza di struct validate campo-per-campo, con
  errori chiari (lista/membro mancanti, rate ≤ 0).

## 10. Export — issue #5, #87 (fix review I1, minors)
`internal/export`, più flag headless (§11):
- **HTML** self-contained: template Go puro, **CSS inline**, **nessun asset/font
  remoto** (vietato esplicitamente; il golden test lo blocca). Mostra periodo+tz,
  split billable, `CurrencySubtotals`, budget, summary line. Stampabile in PDF dal
  browser.
- **CSV invoicing** (#87): profilo colonne comune per tool di fatturazione
  (Fatture in Cloud/QuickBooks…): `date, list_id, client, description, qty_hours,
  rate, amount, currency, billable`. Una **riga per unità del ledger** (§5.3) →
  `sum(amount) == CurrencySubtotals`. Colonne documentate nei README.
- Caveat multi-tag (§11) mostrato **anche** negli export per-tag, non solo in TUI.

## 11. Tag come categorie — issue #53 (ridotta)
- Il **filtro per tag esiste già** (`FilterCriteria.Tags`) e `TimeEntry.Tags` è già
  popolato → **niente nuovo endpoint** `time_entries/tags` (fix I4, assunzione API
  in meno).
- Si aggiunge `GroupByTag`: l'elenco tag si **deriva dalle entry caricate**. Un'entry
  multi-tag conta in **ogni** bucket-tag → i subtotali per-tag possono eccedere il
  totale unico: **caveat esplicito** in TUI e negli export.
- **Filtro lista name-based (limite dichiarato):** `Filter` matcha la lista per
  `ListName` (nome), quindi liste omonime matchano entrambe un nome selezionato — limite
  accettato e documentato. `Build` invece **bucketizza e calcola gli importi per ID**
  (correttezza valuta): le due cose sono indipendenti. Il filtro billable è
  `FilterCriteria.Billable *bool` (nil = tutte), definito una volta sola e condiviso da
  TUI e headless.

## 12. Client ClickUp — letture nuove, con spike
- `billable` per time entry: decodifica come `*bool`; assente ⇒ `Billable=true`
  (retro-compat: oggi si fattura tutto). Quando il default scatta, il report segnala
  "billable assumed" (non tace). **Spike** (senza rete, da documentare nel piano):
  verificare cosa arriva su workspace dove la ClickApp billable è disattivata.
- Nessun altro endpoint nuovo (tags tagliato, #68 fuori).

## 13. Formatting — issue #83
- `duration.FormatHours(d time.Duration) string` → `X.XXh` (puro).
- Summary line ovunque (UI + export): `N entries · X.Xh · €Y billable`, dove l'importo
  usa la valuta del subtotale (o, multi-valuta, elenca i subtotali).
- Sostituisce il rendering `time.Duration` grezzo (niente più `1h30m0s`).

## 14. Costruttore condiviso — fix review I7
Un solo `PricingFromConfig(cfg config.Config) (report.Pricing, error)` in un punto
condiviso (**`internal/service`** — naturale; `config` che importa `report` sarebbe
lecito ma teniamo la dipendenza in `service`). TUI e headless lo usano entrambi: si
elimina il mirror manuale ammesso oggi in `cli/report.go`. Analogo helper per i
`budgets` e la `*time.Location`.

## 16. Headless — flag & JSON
`clup report` guadagna: `--week <YYYY-Www>` (alt. a `--month`), `--billable`
(filtro), `--tag <name>` (ripetibile), `--group tag`, `--format html|csv-invoice`,
`--tz <IANA>`. Default timezone headless = **UTC** (§6.2). JSON: campi nuovi
**additivi**, più `schema_version` e `timezone` nell'oggetto. Lo schema resta
"stabile" (parsabile con `jq`); i parser esistenti non si rompono.
