# Design — v1.6 "Rebrand & foundations"

> Spec di progetto. Documento di design in italiano (convenzione `docs/superpowers/`).
> Tutto ciò che finisce nel codice (identificatori, commenti, stringhe UI, test) va in **inglese**.

## Obiettivo

Posare le fondamenta tecniche su cui poggiano tutte le milestone successive, e ribrandizzare
il binario `clickup` → `clup` in sicurezza. Otto issue in un unico spec/branch/PR:

- **#34** Rebrand `clickup` → `clup` (migrazione config sicura).
- **#35** Estrarre `internal/service` (pipeline dati fuori dalla TUI).
- **#36** Unificare il core HTTP del client (base per i verbi di scrittura).
- **#37** Rate limiter globale + `Retry-After` + retry su tutti i verbi.
- **#38** `retryableErrMsg` (retry-with-context: l'errore API non perde il form).
- **#39** Clock injection sul root `Model`.
- **#40** Endpoint Get Filtered Team Tasks.
- **#41** Headless `clup report --json` (con cobra).

Epic roadmap: [#33](https://github.com/marcoarnulfo/clickup-cli/issues/33).

## Contesto (stato attuale, verificato sul codice)

- `cmd/clickup/main.go` (12 righe): `config.Load()` → `tea.NewProgram(tui.New(cfg), tea.WithAltScreen())` → `Run()`. Nessun framework CLI.
- `internal/config/config.go`: `Config{Token, WorkspaceID, Currency, Rate float64, Rates map[string]float64}`. `Path()` = `filepath.Join(os.UserConfigDir(), "clickup-cli", "config.yml")` (hardcoded, **non iniettabile**). `Load()` legge il file (assente → `Config{}` senza errore), poi `CLICKUP_TOKEN` sovrascrive `Token`. `Save()` = `MkdirAll` + `WriteFile` 0600.
- `internal/clickup/client.go`: `Client{token, BaseURL, http, mu, listNames}`. Solo `get`/`post`. `getRetry` ritenta su 429 con `retryDelay=2s` (var, override nei test), `maxRetries=2`, **solo GET**, ignora `Retry-After`. `finishJSON` centralizza status+decode; 401 → `ErrUnauthorized`. Auth `Authorization: <token>` (NO Bearer). `ListNames` (list.go) già parallelizza con bounded-concurrency (semaforo a canale + WaitGroup + mutex sulla mappa).
- `internal/tui/app.go`: root `Model` value-receiver (~50 campi, ~12 schermate, `screen` = iota). La pipeline report (`TeamMembers` → `TimeEntries` → enrichment `ListNames` → `report.Build`) vive **inline** in `loadEntriesCmd`/`reloadEntriesCmd` + handler `entriesMsg`. `statusEnrichCmd` fa **N `GET /task/{id}` sequenziali**. `currentRange()` chiama `time.Now()` e codifica la regola "custom end inclusivo, +1 giorno". `errMsg` generico → `screenError` (perde il form); `logErrMsg` già preserva il form del log. `reloadEntriesCmd` ha un ramo demo (il demo **non fa I/O**). `q` esce tranne su setup/rates/range (hardcoded).
- `internal/report`/`internal/duration`: **puri** (nessun `time.Now` interno; i preset prendono `now` esplicito).
- `internal/export`: `CSV`/`JSON`/`Markdown` scrivono su `io.Writer`. `JSON` è indentato, snake_case.

## Decisioni bloccate (non ridiscutere)

1. v1.6 = **un solo** spec/branch/PR (le 8 issue insieme), ma ogni step è un commit verde e reviewabile.
2. Headless CLI su **cobra**.
3. Rebrand `clickup` → `clup` con **fallback di lettura config permanente** + shim `clickup` deprecato per 2-3 release.
4. Le review fable (roadmap + design v1.6) sono incorporate qui sotto.

## Architettura risultante

```
cmd/clup/main.go       → internal/cli.Execute()          (nuovo binario)
cmd/clickup/main.go    → internal/cli.Execute(); dopo p.Run() stampa la deprecation
internal/cli           → cobra root (Use: "clup"):
                           • root, Args=NoArgs, nessun subcomando → lancia la TUI
                           • `report` subcomando → headless
internal/service       → LoadEntries(...) : la pipeline I/O (oggi in tui/app.go)
internal/clickup       → do(method,…) unificato (limiter+retry) + FilteredTeamTasks
internal/config        → path clup + fallback legacy + schema_version + fix token-env
internal/tui           → Model.now func; retryableErrMsg (schermate con form)
internal/report        → regola custom-range +1g (pura) spostata qui
```

`internal/cli` **nasce nello step service** e vi si ripunta subito il vecchio `cmd/clickup`; lo
step rebrand poi rinomina soltanto la directory, aggiunge lo shim e cambia i path di config.

## Componenti per package

### `internal/clickup` — core HTTP unificato (#36, #37)

- Rifattorizzare `get`/`post` in un unico:
  `do(ctx, method, path string, query map[string]string, body, out any) error`.
  - Prima di **ogni tentativo** attende `c.limiter.Wait(ctx)`.
  - Setta `Authorization` (no Bearer); `Content-Type: application/json` se c'è body.
  - Su **429**: legge `Retry-After` (delta-secondi via `strconv.Atoi` **oppure** HTTP-date via `http.ParseTime`), altrimenti backoff (`retryDelay` attuale); **clampa** l'attesa a min(~30s, ctx residuo); ritenta fino a `maxRetries`, **il retry riconsuma il limiter**. Retry **solo su 429**, mai 5xx/transport. Vale per **tutti** i verbi (429 = richiesta rifiutata, non processata).
  - Poi `finishJSON`.
- `get`/`post` diventano wrapper sottili su `do`. **NON** aggiungere `put`/`delete` ora (nessun chiamante in v1.6 → sarebbero dead-code U1000): arriveranno col primo chiamante (v1.8). `do` con `method` string rende l'aggiunta un diff di 3 righe.
- Limiter sul `Client`: `c.limiter = rate.NewLimiter(rate.Limit(90.0/60.0), 30)` (≈1.5 req/s, burst 30). **Test seam**: campo/var iniettabile impostato a `rate.NewLimiter(rate.Inf, 0)` nei test (come `retryDelay`), così le suite httptest esistenti non rallentano a secondi reali sotto `-race`. Mantenere un hook equivalente all'attuale `retryDelay` var, o i test dei retry dormono 2s.
- Rivedere i timeout ctx a monte (`loadEntriesCmd` 30s, `statusEnrichCmd` 60s): con il limiter l'attesa erode quei budget; alzarli/renderli coerenti nello stesso step per non introdurre timeout su report grandi.

Dipendenza nuova: `golang.org/x/time/rate`.

### `internal/clickup` — Get Filtered Team Tasks (#40)

- `FilteredTeamTasks(ctx, teamID string, f TaskFilter, page int) ([]Task, bool, error)` su `GET /team/{id}/task` (100/pagina). `TaskFilter` copre il sottoinsieme utile ora (`statuses`, `assignees`, `list_ids`, `tags`, date), estendibile.
- Il bool "hasMore" è **inferito** (la v2 restituisce `{"tasks":[...]}` senza metadati di pagina): ultima pagina = pagina corta/vuota. Pinnare con un httptest che restituisce **esattamente 100** task su page 0.
- Helper `AllFilteredTeamTasks(ctx, teamID, f)` che itera le pagine.
- **NON** cablare questo endpoint dentro `statusEnrichCmd` in v1.6 (vedi sotto): questo endpoint chiude #40 come *fondamenta* per la search di v1.10.

### `internal/tui` — statusEnrich parallelo (parte di #40, correzione review)

- `statusEnrichCmd` resta **per-task** ma diventa **parallelo con bounded-concurrency**, riusando il pattern di `ListNames` (semaforo a canale bufferizzato + `WaitGroup` + mutex sulla mappa risultati). Nessuna nuova semantica API, nessun rischio su closed/subtasks/archiviati, nessuna inversione di costo.
- Tollera errori per-task come oggi (status `""`, non abortisce; abort solo su `ErrUnauthorized`).
- Motivazione (review): `GET /team/{id}/task` di default **esclude closed/subtasks/archiviati** → in un report di fatturazione mancherebbe la maggior parte dei task; e su una lista con migliaia di task il batch pagina O(task totali) contro O(task distinti del report) del per-task. Lo swap batch è quindi **fuori scope** di v1.6.

### `internal/report` — regola di range (correzione review, #35/#41)

- Spostare la regola "custom end **inclusivo**, `end = to + 1 giorno`" da `Model.currentRange()` a una funzione **pura** in `internal/report` accanto a `RangeForPreset` (es. `CustomRange(from, to time.Time) (start, end time.Time)`).
- Entrambi i frontend (TUI e headless) la usano → nessuna divergenza sui giorni di confine.
- **Timezone pinnata**: i confini (`--month`, `--from/--to`, preset) in **UTC** (mezzanotte UTC), **coerente col codice reale della TUI** (`RangeForPreset` usa `midnightUTC`; le date custom sono parsate con `time.Parse("2006-01-02", …)` → UTC). NON usare l'ora locale nell'headless, o headless e TUI divergono sui giorni di confine.

### `internal/service` — estrazione pipeline (#35)

- `service.LoadEntries(ctx, c *clickup.Client, teamID string, start, end time.Time, scope string, assignees []int) ([]report.TimeEntry, error)` = la **parte I/O** oggi in `loadEntriesCmd`:
  - se `scope == "team"` e `assignees` vuoto → espande via `TeamMembers` (l'espansione vive **nel service**, non duplicata nei due chiamanti);
  - `clickup.TimeEntries(...)` (già con `include_task_tags`);
  - enrichment nomi lista.
- **`report.Build` resta puro e resta nei chiamanti**: la TUI ricostruisce localmente su cambio grouping/filtri; l'headless chiama `Build` una volta. Il service restituisce le **entries**, non il Report.
- Il **demo resta solo TUI** (il service non tocca il demo): `reloadEntriesCmd` mantiene il ramo demo e chiama `service.LoadEntries` solo nel ramo reale, che si riduce a un wrapper ~10 righe che ritorna `entriesMsg`/`errMsg`.
- Status enrichment: **rimane TUI-only** in v1.6 (l'headless non fa `--group status`). Dichiararlo nello spec per non generare rework.

### `internal/cli` — cobra root + headless (#41)

- `internal/cli.Execute()` costruisce il root cobra `Use: "clup"`, `Args: cobra.NoArgs`, `SilenceUsage: true`, `SilenceErrors: true`; l'errore è stampato **una volta** in `Execute()`.
- Root senza subcomando → lancia la TUI (`config.Load` → `tui.New(cfg)` → `tea.NewProgram(..., WithAltScreen)` → `Run`), **rispettando `CLICKUP_DEMO`** come oggi.
- Subcomando `report`: flag `--month YYYY-MM` | `--from`/`--to` | preset (`--preset`), `--scope me|team`, `--group`, `--format json|csv|md` (default `json`). RunE: `config.Load` → costruisci client → risolvi range (`report.RangeForPreset`/`CustomRange`) → `service.LoadEntries` → `report.Build` → `export.<Format>(os.Stdout, r)`. **Nessun bubbletea** quando c'è un subcomando. Tutti gli export sono single-stream su stdout (help: "usa la redirezione della shell").
- **Failure UX headless** (l'headless non ha wizard):
  - config non valida / token assente → stderr `run 'clup' to configure, or set CLICKUP_TOKEN`, exit **1**;
  - `ErrUnauthorized` a metà run → messaggio analogo, exit 1;
  - `CLICKUP_DEMO=1 clup report` → **ignora** l'env demo, procede con la config reale (documentato); il demo è solo TUI.
- `cmd/clup/main.go` → `internal/cli.Execute()`. Nello step service si **ripunta** già `cmd/clickup/main.go` a `internal/cli.Execute()` (evita un secondo main ad-hoc).

### `internal/tui` — clock injection (#39)

- `Model.now func() time.Time`; `New(cfg)` lo imposta a `time.Now`. `currentRange()` usa `m.now()`. Seam per i test (`NewWithClock` o campo settabile) per iniettare un clock fisso.
- `report` resta puro (riceve `now` esplicito da service/headless/TUI).

### `internal/tui` — retryableErrMsg (#38)

- `retryableErrMsg{origin screen, err error}`. Le cmd delle **schermate con form/selezione** (filters, members, range) catturano l'origine al dispatch e, su errore, ritornano `retryableErrMsg` invece di `errMsg` generico.
- Handling: `ErrUnauthorized` → wizard (come ora); altrimenti **preserva il modello della schermata di origine** e mostra l'errore lì (niente switch a Home), generalizzando ciò che `logErrMsg` già fa per il log.
- **Ambito ridotto** (review): solo schermate con stato in-progress; Home→Loading→errore non perde nulla e resta su `errMsg`/`screenError`. Niente auto-retry command in v1.6 (rimandabile) → dimezza il churn su `app_test.go`.

### `internal/config` — path clup + migrazione (#34)

- `Config` acquisisce `SchemaVersion int` (`yaml:"schema_version"`). State machine: campo assente → **0** (legacy), corrente → **1**; `migrate(c) Config` è una funzione **totale e idempotente** `if v == 0 { v = 1; … }`; file con versione **> corrente** → warning e prosegui. Niente registry/chain (arriva a v1.10 se serve).
- `Path()` → `filepath.Join(os.UserConfigDir(), "clup", "config.yml")`; `legacyPath()` → `…/clickup-cli/config.yml`. **Reso iniettabile** per i test (`t.TempDir`, e per testare il fallback servono entrambi i path override).
- `Load()`: legge il nuovo path; se assente, legge il **legacy**; `CLICKUP_TOKEN` override invariato.
- **Migrazione one-shot** (quando si legge dal legacy e il nuovo è assente): al primo `Save()` si scrive il nuovo path e si **riscrive il file legacy a stub-puntatore** (`# moved to <new path> by clup`) — soddisfa "non cancellare" ma uccide lo split-brain "edito il vecchio, non succede nulla". (Niente notice runtime: lo stub è di per sé il segnale; evita di far stampare `Save`, che è una libreria.)
- **Fix bug latente token-env + seam di stato**: `Load()` sovrascrive `Token` da `CLICKUP_TOKEN`, e un `Save()` successivo (la schermata rates salva) **persisterebbe il token env su disco** — la migrazione lo peggiora. Seam concreto: `Config` acquisisce due campi **unexported** (non serializzati da yaml per definizione) `fileToken string` e `loadedFromLegacy bool`, settati da `Load`. `Save` lavora su una copia: se `env := os.Getenv("CLICKUP_TOKEN"); env != "" && c.Token == env` → `copy.Token = c.fileToken`; se `c.loadedFromLegacy` e il nuovo file non esiste ancora → scrive lo stub sul legacy. (Edge accettabile: se il wizard digita un token identico all'env, resta il file token.)
- **Path per-OS**: `os.UserConfigDir()` su macOS è `~/Library/Application Support`, non `~/.config`. Documentare i path reali per-OS in README (niente affermazioni false su `~/.config`).

### Rebrand: repo-wide (#34)

- Rinominare `cmd/clickup/` → `cmd/clup/` (binario `clup`; install `…/cmd/clup@latest`).
- `cmd/clickup/` diventa **shim deprecato**: chiama `internal/cli.Execute()` e stampa su **stderr** la deprecation **dopo** `p.Run()` (per il path TUI l'AltScreen la cancellerebbe se stampata prima) e prima/subito per il path subcomando.
- **Task di grep del nome binario su tutto il repo** e aggiornamento: `docs/demo.tape` (VHS invoca il binario per nome), `.github/workflows/*`, eventuale config release/Homebrew, righe `go install`/`go build` e path config in **entrambi** i README (con la nota fallback e i path per-OS), `CLAUDE.md`. Il **module path** `github.com/marcoarnulfo/clickup-cli` resta invariato.

## Gestione errori

- Client: `do` distingue 401 (`ErrUnauthorized`), 429 (retry con clamp), altri non-2xx (errore formattato) come oggi.
- TUI: `ErrUnauthorized` → wizard; errori su schermate con form → `retryableErrMsg` (form preservato); resto → `screenError`.
- Headless: nessun wizard; errori → stderr + exit 1 (vedi sopra).

## Testing (TDD, table-driven)

- `internal/clickup`: `do` via httptest — happy path GET/POST, 429 con `Retry-After` **secondi** e **HTTP-date** + clamp, retry che riconsuma il limiter, limiter iniettato a `rate.Inf` nelle suite esistenti; `FilteredTeamTasks` paginazione (page da 100 esatti → hasMore, page corta → fine); errori non-2xx.
- `internal/report`: `CustomRange` (end inclusivo/+1g, timezone), invarianza con i preset esistenti.
- `internal/service`: `LoadEntries` via httptest (scope me/team, espansione assignees, enrichment nomi lista).
- `internal/config`: `Load` con nuovo/legacy/assente (`Path` iniettabile, `t.TempDir`), migrazione one-shot + stub legacy + `schema_version` state machine, e il **non-persistere il token env**.
- `internal/cli`: `report` RunE contro un server fake → asserisce lo **stdout** (json/csv/md); no-token → stderr + exit 1; `Args: cobra.NoArgs` (arg sconosciuto → errore, non TUI).
- `internal/tui`: `statusEnrichCmd` parallelo (mock N task, tolleranza errori per-task); `retryableErrMsg` (form preservato, ritorno all'origine, `ErrUnauthorized` → wizard) via `Update()`+msg; clock iniettato in `currentRange`.
- `internal/report`/`internal/duration` restano **puri** (nessun import I/O).

## Demo mode

- Invariato: la TUI in `CLICKUP_DEMO=1` non fa I/O; il ramo demo di `reloadEntriesCmd` non passa dal service. L'headless ignora l'env demo (documentato).

## Invarianti / composizione

- Purezza di `internal/report`/`internal/duration` mantenuta; la regola di range diventa un'altra funzione pura lì.
- Il seam I/O (service) è l'unico punto che orchestra le chiamate; `Build` resta puro e nei chiamanti.
- La TUI conserva il suo comportamento (schermate, keybinding) tranne `retryableErrMsg` e clock.
- **Contratto JSON headless "congelato"**: `export.JSON` diventa una API di scripting; cambiarne lo schema in futuro rompe le pipeline `jq` degli utenti → trattarlo come stabile (eventuale versionamento in seguito).

## Fuori scope / rimandato

- Verbi `put`/`delete` esportati (arrivano col primo chiamante, v1.8).
- Swap batch di `statusEnrichCmd` su `FilteredTeamTasks` (rischio correttezza; resta il per-task parallelo).
- Auto-retry command in `retryableErrMsg` (solo preservazione del form ora).
- Cache offline, profili, keychain (milestone successive).
- Migration registry/chain (arriva se e quando servirà, v1.10).

## Note dalle review (da onorare in fase di piano)
