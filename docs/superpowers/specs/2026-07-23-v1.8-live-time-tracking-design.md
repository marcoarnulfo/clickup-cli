# v1.8 — Live time tracking — Design

> Spec for the v1.8 milestone of `clickup-cli`. Covers issues #91 (live ticking
> timer), #94 (edit a time entry), #98 (delete a time entry), #100 (time-entry
> change history). Branch: `v1.8-live-time-tracking`.

## 1. Goal

Turn the existing (but static and hidden) timer into a **live, always-visible**
tracker, and add a **time-entry browser** from which a user can **edit**,
**delete**, and **inspect the history** of their logged entries — all with full
demo-mode parity and without breaking the purity of `internal/report` /
`internal/duration`.

## 2. Scope

**In:**
- #91 Live ticking timer: 1s stopwatch (`HH:MM:SS` + `(X.XXh)`), a global
  running-timer indicator on Home, ticking on the timer screen.
- #94 Edit a time entry: core fields (duration, start, description, billable).
- #98 Delete a time entry: with a confirmation dialog.
- #100 Time-entry change history: read-only view (spike-first; **sacrificial** —
  see §9).

**Out (follow-up issues):**
- Editing an entry's **tags** (needs a space-tag fetch + multi-select picker).
  A new issue is filed for it and referenced from #94.

## 3. Architecture & boundaries

- **`internal/clickup`** — three new verbs, all through the existing method-agnostic
  `do()` (its 429 retry is safe for these; PUT/DELETE are idempotent):
  - `UpdateTimeEntry(ctx, teamID, entryID string, start time.Time, dur time.Duration, desc string, billable bool) error` → `PUT /team/{id}/time_entries/{id}`
  - `DeleteTimeEntry(ctx, teamID, entryID string) error` → `DELETE /team/{id}/time_entries/{id}`
  - `TimeEntryHistory(ctx, teamID, entryID string) ([]HistoryChange, error)` → `GET /team/{id}/time_entries/{id}/history`
  - The client stays **clock-free**: elapsed time is never computed here.
- **`internal/duration`** — stays pure; gains one pure helper:
  - `Clock(d time.Duration) string` → `"01:23:45"` (zero-padded H:MM:SS; hours
    are not capped at 24). Reuse the existing `FormatHours` for the `"1.40h"`
    part — **do not** redefine it.
- **`internal/report`** — untouched. It already carries everything the browser
  needs on `TimeEntry` (`ID`, `UserID`, `TaskName`, `Start`, `Duration`,
  `Billable`, `Tags`).
- **`internal/tui`** — the bulk of the work: a new `screenEntries` (browser +
  edit form + delete confirm + history view), a tick loop, a Home indicator, the
  `userID` on the Model, and demo session-state for edit/delete. Elapsed is
  derived here from the injected clock `m.now()`.

**Purity guard:** no new import of config/clickup/tui inside report/duration.
`Clock` is a string formatter with no clock and no I/O.

## 4. Live timer (#91)

### 4.1 Running-timer state on the Model
- Add `m.runningTimer *clickup.RunningTimer` (nil = none running) and
  `m.ticking bool` (whether a 1s tick chain is currently armed).
- Add `m.userID int` (authenticated user, for ownership gating — §6).

### 4.2 Startup probe (distinct message — do NOT reuse `timerMsg`)
`Init()` currently returns a single cmd (`updateCheckCmd`). It becomes:

```go
func (m Model) Init() tea.Cmd {
    return tea.Batch(m.updateCheckCmd(), m.runningTimerProbeCmd(), m.currentUserCmd())
}
```

- `runningTimerProbeCmd` returns a **new** `runningTimerMsg{timer *clickup.RunningTimer}`
  (real: `CurrentTimer`; demo: a fake running timer with a **non-zero** Start,
  see §4.5). It is separate from `timerMsg` on purpose: `timerMsg` is guarded
  (`app.go:620`, dropped unless screen is Log/Loading) because it drives the log
  timer flow. The global probe fires while screen == Home, so it must not be
  dropped.
- `currentUserCmd` returns `userMsg{id int}` (real: `CurrentUser`; demo:
  `demoSelfID`). Failure/latency → `m.userID` stays 0 (gating handles it, §6).

### 4.3 `runningTimerMsg` handler (split responsibilities)
The handler **always** updates global state and arms the tick; it **never**
changes the screen:

```go
case runningTimerMsg:
    m.runningTimer = msg.timer
    if msg.timer != nil && !m.ticking {   // arm only on nil -> non-nil
        m.ticking = true
        return m, tickCmd()
    }
    if msg.timer == nil {
        m.ticking = false                 // let any in-flight tick chain die
    }
    return m, nil
```

### 4.4 Tick loop (no idle ticks, no double-arming)
- `tickCmd()` = `tea.Tick(time.Second, func(time.Time) tea.Msg { return tickMsg{} })`.
  (The tick reads no wall-clock value from the callback — elapsed is derived from
  `m.now()` in the view, keeping the clock injectable/testable.)
- `tickMsg` handler:

```go
case tickMsg:
    if m.runningTimer == nil {
        m.ticking = false
        return m, nil     // stop the chain: no timer, no ticks
    }
    // periodic re-poll so a timer started/stopped in another ClickUp client
    // is reflected here; cheap, well within the rate limiter.
    m.tickCount++
    if m.tickCount%30 == 0 {
        return m, tea.Batch(tickCmd(), m.runningTimerProbeCmd())
    }
    return m, tickCmd()
```

- **Double-arming guard:** arming happens only on the nil→non-nil transition in
  the `runningTimerMsg` handler *and* the tick chain re-issues itself only from
  the `tickMsg` handler. Starting a timer from the log flow routes through
  `runningTimerMsg` (see §4.6), so there is exactly one chain. `m.ticking`
  prevents a second chain if a probe arrives while one already runs.

### 4.5 Elapsed rendering (zero/negative Start guards)
Shared pure helper drives both Home and the timer screen:

```go
// elapsedLabel returns "01:23:45  (1.40h)" for a running timer, or "" when the
// start is unknown (zero) — three code paths yield a zero Start, so this MUST
// be guarded (a zero Start would otherwise render a ~490,000h stopwatch).
func elapsedLabel(start, now time.Time) string {
    if start.IsZero() {
        return ""
    }
    d := now.Sub(start)
    if d < 0 { // server clock ahead of client: clamp, never show negative
        d = 0
    }
    return duration.Clock(d) + "  (" + duration.FormatHours(d) + ")"
}
```

- `Start` is UTC (`time.UnixMilli(ms).UTC()`); `now.Sub(start)` is instant-based
  and location-independent — **no timezone bug**, `m.loc` is irrelevant to elapsed.
- Demo fake timers must set a **non-zero** Start relative to `m.now()` (e.g.
  `m.now().Add(-25*time.Minute)`) so the ticking is visible.

### 4.6 Home indicator
- When `m.runningTimer != nil`, Home shows a live line below the existing
  update-notice pattern:

```
⏱  running on Fix login bug — 01:23:45  (1.40h)   (c: manage)
```

- Key **`c`** on Home (not `t`, which toggles me/team; not shift-`T`, adjacent
  and confusing) jumps to the timer screen with origin = Home. When no timer is
  running, `c` does nothing and the line is absent.
- `home.view` gains the running timer and `m.now()` (or a pre-rendered line) as
  arguments. The help line stays readable.

### 4.7 Timer screen (`logTimerRunning`)
- Replaces the current static display. Renders `elapsedLabel(timer.Start, m.now())`
  live; if it returns `""` (zero Start), show `"started just now"` instead of a
  bogus stopwatch.
- Keys unchanged: `s` stop and record, `Esc` back — but **back respects origin**
  (§4.8).

### 4.8 Origin for the timer/log flow
- The log flow gains an `origin screen` field (mirroring
  `listBrowserModel.origin`). `logTimerRunning` `Esc`, and the `logDone` step
  `Esc`/`Enter`, return to `origin` (Home if entered via `c`, else Report as
  today). This stops a `c`-from-Home user from landing on a zero-value
  `reportModel`.

### 4.9 Stopping the timer clears the indicator (dedicated message)
- `stopTimerCmd` must emit a **new** `timerStoppedMsg{summary string}` — **not**
  the shared `logDoneMsg` (also emitted by `createEntryCmd`; clearing the
  indicator on every `logDoneMsg` would kill a running timer's indicator when the
  user merely logs a manual entry).
- `timerStoppedMsg` handler: set `m.runningTimer = nil`, `m.ticking = false`,
  route to the `logDone` step with the summary, and (real mode) the entry now
  exists so the report is stale — reload is offered as today (`r` on logDone).

## 5. Entries browser (`screenEntries`)

### 5.1 Entry & wiring
- New screen constant `screenEntries`, **appended at the END** of the `screen`
  iota block (per the no-reorder rule).
- Entered from the **Report** screen via key **`v`** (view/entries). `e` is
  taken by Export; `v` is free. Add `v: entries` to the Report help line.
- Add `screenEntries` to the global `q`-quit exception list (`app.go:532`) so
  free-text input in the edit form can contain `q`. **Also add the pre-existing
  `screenLog`** to that list in the same change (its note field has the same
  latent bug today).

### 5.2 Data source
- The browser renders **`m.visibleEntries()`** (filter-applied), **not**
  `m.entries` (which is pre-filter). Sort a **copy** by `Start` descending; never
  sort `m.entries`/the filter result slice in place (that state feeds
  `report.Build`).
- Sorting helper is a small pure function `sortEntriesByStartDesc([]report.TimeEntry)`
  (table-testable) or inline; running timers (negative duration) are already
  excluded upstream so they never appear here.

### 5.3 Row layout & cursor
```
Entries — Jul 2026 (team)
─────────────────────────
▸ 20 Jul 14:00  Fix login bug      1h30  $   (me)
  20 Jul 09:00  Deploy staging     0h45  $   (bob) — read-only
  19 Jul 11:00  Standup            0h15      (me)
```
- `$` marks billable. Owner shown as `(me)` / `(<username>)`. `— read-only` on
  entries not owned by the user (team scope).
- `↑/↓` / `k/j` move the cursor. Empty state: `"No entries in the current range."`

### 5.4 Actions
- `e` = edit, `x` = delete, `h` = history, `Esc` = back to Report.
- `e` and `x` are **enabled only on the user's own entries** (§6). On a read-only
  entry they show a brief status: `"That entry belongs to <user> — read-only."`
- `h` (history) is allowed on any entry (read-only inspection).

### 5.5 Reload routing (return to the browser, not Report)
- After a successful edit or delete, the browser refreshes so it and the report
  reflect the change — but the user must land **back in the browser**, not on
  Report (the plain `entriesMsg` handler ends on `screenReport`).
- Introduce `entriesReloadedMsg{entries []report.TimeEntry}` handled to: update
  `m.entries`, re-stamp statuses, prune filters, rebuild `m.report` (same as
  `entriesMsg`), then **stay on `screenEntries`** and **clamp the cursor** into
  the new (possibly shorter) visible list. `reloadEntriesCmd` is parameterized by
  origin, or a dedicated `reloadEntriesForBrowserCmd` emits this message.
- If the edited entry moved outside the current range (start changed to another
  day/month), it vanishes from the list after reload — show a status line
  `"Entry saved — it moved outside the current range."` so it never reads as data
  loss.

## 6. Ownership gating (#94/#98 safety)

- `m.userID` is resolved once at startup via `currentUserCmd` (§4.2). Demo uses
  `demoSelfID` (= 1, alice), which matches demo entries owned by uid 1 — do not
  invent a new fake id.
- Predicate: **`canEdit(entry) = m.userID != 0 && entry.UserID == m.userID`**.
  The `userID != 0` guard is required because `rawEntry.User.ID` defaults to 0
  when absent — without it, a zero self-id would spuriously match zero-owner
  entries.
- If `currentUserCmd` failed/hasn't returned when the browser opens (`m.userID
  == 0`), everything is read-only. On browser open, if `m.userID == 0`, re-issue
  `currentUserCmd` (lazy retry).

## 7. Edit a time entry (#94)

- Reuses the input helpers (`newTextInput`), fields filled in sequence:
  **Duration** (`duration.Parse`), **Start** (date `YYYY-MM-DD` + time `HH:MM`),
  **Note**, **Billable** (`[Y/n]` toggle). Prefill from the selected entry's
  current values.
- **Clock discipline:** prefill and parsing use `m.now()` / `m.loc` — **not**
  `time.Now()` / hardcoded `time.Local`. (The existing `enterForm` /log create
  path has that wart at `log.go:208,353`; the edit form must not copy it.)
- On confirm: `UpdateTimeEntry(ctx, teamID, entry.ID, start, dur, note, billable)`.
  Wire format mirrors `CreateTimeEntry` exactly: `start` and `duration` as
  **integer milliseconds**, `billable` bool, `description` string. Always send
  the full core set (start+duration+description+billable), not a partial diff.
- Success → `entriesReloadedMsg` flow (§5.5) with a `"Entry saved."` status
  (or the moved-outside-range variant).
- Errors route through `logErr` (auth → global re-setup; else stay on the screen
  with the message).

## 8. Delete a time entry (#98)

- Destructive → explicit confirmation. From the browser, `x` on an owned entry
  opens a confirm step: `Delete this entry? [y/N]` (default **No**). Only `y`
  confirms; any other key cancels back to the list.
- Confirm → `DeleteTimeEntry(ctx, teamID, entry.ID)` → `entriesReloadedMsg` flow
  with `"Entry deleted."`; cursor clamped.

## 9. Time-entry change history (#100) — spike-first, sacrificial

- **First task of this section is a spike:** confirm the response shape of
  `GET /team/{id}/time_entries/{id}/history` with an `httptest` fixture built
  from a real sample payload. Only then build the view.
- `HistoryChange{Field, Before, After string, Date time.Time, User string}`.
  `Before`/`After` are **normalized to strings** (ClickUp may return objects or
  scalars) with a raw-JSON fallback so the view never breaks on an unexpected type.
- Read-only chronological view opened with `h`; each row `Date · Field: Before → After · User`.
- A history **fetch failure shows inline** in the browser (via `logErr`/status),
  never a dead-end `screenError`.
- **Sacrificial scope:** if the spike shows the endpoint returns nothing useful
  (or no public API), v1.8 ships #91/#94/#98 and #100 is dropped to a follow-up.
  The plan orders history **last** so this cut costs nothing already built.

## 10. Demo-mode parity

- `CLICKUP_DEMO=1` never touches `m.client`. Every new command has a demo branch.
- **Startup probe:** a dedicated demo cmd returns a fake running timer with a
  non-zero Start (§4.5). It must be **separate** from the existing
  `demoCurrentTimerCmd` (which returns nil — load-bearing for the log timer flow,
  where nil means "no timer, pick a task"; a fake timer there would trap demo
  users in `logTimerRunning`).
- **Edit/delete need session state**, because `demoEntries` is a pure function of
  the range and every reload regenerates the fixture (a naive delete+reload would
  resurrect the entry; an edit+reload would discard the change). Add to the Model:
  - `demoDeleted map[string]bool` (entry IDs hidden after a demo delete)
  - `demoOverrides map[string]report.TimeEntry` (edited values by entry ID)
  - The demo entries command applies overrides and filters out deleted IDs before
    returning, so the browser/report reflect demo edits and deletes for the
    session. (Real mode ignores these maps entirely.)
- `demoHistoryCmd` returns a small fixed history so the view is demonstrable.
- The Home ticking line and the new browser change `docs/demo.tape` output — the
  README GIF is regenerated (`vhs docs/demo.tape`) as the final doc step.

## 11. Testing (TDD)

- **Client** (`httptest`): `UpdateTimeEntry` (PUT, path, ms body, `billable`),
  `DeleteTimeEntry` (DELETE, path, 200/204), `TimeEntryHistory` (GET, decode,
  string normalization + raw fallback).
- **duration**: `Clock` table test (zero, sub-minute, hours>24, exact hour).
- **TUI** (transitions via `Update()` + simulated msgs, deterministic injected
  `now` — no real sleeps):
  - `runningTimerMsg` arms exactly one tick chain; nil clears it; no double-arm on
    a second non-nil probe.
  - `tickMsg` stops when `runningTimer == nil`; re-polls on the 30th tick.
  - `elapsedLabel` zero Start → `""`; negative → `00:00:00 (0.00h)`.
  - Browser: `v` from Report opens it; ownership gating (`canEdit` predicate);
    `x` confirm `[y/N]` (only `y` deletes); reload returns to `screenEntries` with
    a clamped cursor; edit-moved-outside-range status.
  - `timerStoppedMsg` clears the indicator; `logDoneMsg` (manual create) does not.
  - `q` typeable in the edit note (screenEntries in the exception list).
  - Origin: `c` from Home → timer screen → `Esc` returns to Home.
  - Demo: delete hides an entry across reload; edit persists across reload;
    demo startup probe shows a ticking timer; `canEdit` true for demo self-owned
    entries.
- **Gate before every commit:** `gofmt -l .`, `go vet ./...`, `go build ./...`,
  `go test ./... -race` (all clean/green).
